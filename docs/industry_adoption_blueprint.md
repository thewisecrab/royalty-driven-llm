# Industry Adoption Blueprint

This project should become more than a prototype. The path to industry adoption is
to make attribution unavoidable, cheap, verifiable, and useful to everyone in the
chain.

## North Star

Every foundation-model response that uses owner-controlled content should produce:

1. a grounded answer with visible source labels
2. an attribution receipt
3. a creator settlement record
4. a transparency-log inclusion proof
5. a rights-policy decision trail
6. an ownership-registry decision trail
7. claim-level evidence span hashes
8. a grounding-quality verdict and component scores
9. a verifiable escrow-release report when a dispute is resolved
10. a portable VC/PROV-style interoperability bundle
11. an OpenTelemetry-aligned trace exchange for source access, citation, and claim
    support
12. a periodic royalty statement rollup for creator and escrow accounting
13. a creator challenge and correction report when attribution is contested
14. a derivative lineage report that preserves upstream creator obligations through
    summaries, synthetic datasets, tool outputs, and transformed corpora
15. a provider attribution card for model-level certification and coverage disclosure
16. a training-content summary for model-level corpus and rights transparency
17. a user-facing answer provenance card that binds footer labels and claim spans to
    receipts and traces
18. a source verification report proving cited sources and claim evidence spans
    materialize from registered content hashes without private text disclosure
19. a source confidence report exposing verified/warning/failed footer rows, claim
    confidence rows, and hallucination taxonomy for fabricated sources, drift,
    license gaps, unsupported claims, and evidence-span gaps
20. a citation footer contract that binds the exact rendered source rows, claim
    anchors, confidence labels, license status, royalty status, display order, and
    footer hash before client display
21. a response envelope that packages rendered answers with embedded public proof
    artifacts at the model API boundary
22. an integration profile that publishes the API contract, schemas, verifier
    commands, public surfaces, readiness checks, and proof hashes
23. a well-known discovery manifest that tells customers and auditors where to find
    and verify every public RDLLM proof surface
24. a public assurance bundle that binds proof-pack artifact hashes and inclusion
    proofs
25. a semantic text attribution report for paraphrased, summarized, or external
    outputs with public source footer rows and escrow for ambiguity
26. a cross-provider attribution exchange manifest and conformance vector pack for
    downstream relay and independent compatibility testing
27. a runtime federation handshake that negotiates minimum attribution level,
    artifact hashes, source-footer relay, escrow relay, runtime headers, signatures,
    and downgrade rejection before provider-to-provider use
28. a portable attribution capsule that keeps the answer's output hash, source
    proof chain, copy marker, C2PA-compatible assertion, and SCITT-like statement
    subject attached when content is copied, reposted, exported, or embedded in
    generated-asset metadata
29. a private audit export when legally authorized
30. a transitive attribution report when copied RDLLM output is used downstream, so
    original source rows and pass-through settlement survive reposting, summaries,
    agent handoff, and model-to-model reuse
31. a clearinghouse report that merges provider royalty statements and transitive
    attribution reports into payable, escrow, and held rows for duplicate-safe
    cross-provider settlement
32. a remittance report that turns cleared payable and escrow rows into
    instruction-only payment rows with payout-account hashes, reconciliation IDs,

33. a payment execution report that reconciles remittance rows against hash-only

34. a payment rail attestation that requires registered external payment or escrow

35. a creator payout receipt report that lets each creator verify paid, escrowed,

36. a rendered attribution audit that proves the exact visible Markdown answer,
    inline source markers, source footer rows, and claim-evidence rows replay from

37. a training-memory provenance report that detects registered source spans
    reproduced from model memory in the exact rendered answer and requires visible

38. a revenue allocation report that proves billing, advertising, subscription,
    API, enterprise, or marketplace revenue pools were conserved into event-level

39. a finance ledger attestation that proves those revenue pools reconcile to
    hash-only external finance exports without disclosing customer, invoice,
    payment, prompt, output, or source text
40. a proof dependency graph that gives auditors a cycle-checked replay order for

41. a persistent memory provenance receipt that treats stored assistant, agent, or

42. a private reasoning attribution receipt that keeps chain-of-thought private

43. a publication monitor that checkpoints public proof surfaces over time, detects

44. a publication witness report that proves the current monitor checkpoint was

57. a universal provider conformance runner receipt that proves official

## Adoption Stages

### Stage 1: SDK Integration

Model providers and RAG vendors add the receipt generator to API responses. This is
the easiest integration because it does not require changing base-model training.

Required outputs:

- source footer
- claim evidence span hashes
- grounding-quality score
- registry status for matched works
- receipt hash
- public receipt endpoint
- private audit export
- transitive attribution report for copied-output reuse
- clearinghouse report for cross-provider settlement
- remittance report for payment-file-ready instruction rows
- payment execution report for processor/escrow settlement reconciliation
- payment rail attestation for registered processor batch signatures
- creator payout receipt report for creator-facing paid, escrowed, or held status
- rendered attribution audit for the exact displayed answer and footer
- training-memory provenance report for memorized registered text in the displayed answer
- persistent memory provenance receipt for stored assistant/agent memory reused in answers
- private reasoning attribution receipt for hidden scratchpads, delegation, and router synthesis
- revenue allocation report for pre-settlement event economics
- finance ledger attestation for hash-only external revenue-source reconciliation
- proof dependency graph for auditor replay ordering and cycle rejection
- publication monitor for append-only proof-surface drift detection
- publication witness report for anti-equivocation over monitor checkpoints
- trust registry for signer and witness key rotation/revocation
- conformance verifier output
- trace-exchange export for source access and citation spans
- royalty-statement export for aggregate settlement periods

### Stage 2: Enterprise Procurement

Enterprises require attribution receipts for high-risk or regulated AI answers.
This mirrors the way SBOMs moved from optional metadata to procurement evidence.

Required outputs:

- audit retention policy
- transparency log roots
- tenant-private prompt/output commitments
- per-source legal basis
- per-source allowed/prohibited use decisions
- per-source ownership-claim status
- public assurance bundle for procurement review
- publication monitor checkpoints for ongoing vendor surveillance
- independent publication witness quorum for split-view detection
- trust registry showing active and revoked provider/auditor/witness keys

### Stage 3: Creator Marketplace

Creators and publishers register works and licenses. AI operators can source from
registered content with usage-based compensation.

Required outputs:

- creator dashboard
- payout statements
- machine-readable license scopes for training, retrieval, generation, display,
  quote, and external attribution
- claim/dispute workflow
- duplicate registration handling with registry-dispute escrow
- post-dispute escrow release reports

### Stage 4: Foundation Model Alignment

Base-model developers publish training cohort receipts and periodic data-value
settlement reports. Per-output training attribution remains statistical, but every
retrieval/tool/output use can be directly receipted.

Required outputs:

- training dataset manifests
- Croissant/SPDX-compatible metadata
- ODRL-compatible permission, prohibition, duty, and constraint terms
- data valuation reports
- public transparency summaries
- grounding-quality aggregate reports for high-risk domains
- provider trace summaries that let customers compare accessed, cited, paid, and
  escrowed source counts across model routes
- foundation API attribution profile at a well-known path so search, chat, agent,
  and downstream model clients can reject attributed answers that omit required
  proof hashes or verifier metadata
- client attribution enforcement receipts from relying clients so downstream
  products cannot silently strip source labels, verifier metadata, or fail-closed
  policy while still claiming attributed output
- composite foundation adapter reports so OpenAI-, Anthropic-, Google-, Meta-,
  Mistral-, Cohere-, xAI-, Bedrock-, Azure OpenAI-, and OpenAI-compatible
  response objects can satisfy one portable attribution contract
- foundation provider conformance matrices so those provider families cannot
  claim compatibility without hash-only positive and fail-closed negative
  fixtures for attribution-critical response modes

### Stage 5: Standardization

The protocol becomes a standards-track artifact aligned with W3C PROV, C2PA,
SCITT, Sigstore/Rekor, SPDX AI, and MLCommons Croissant.

Required outputs:

- JSON schema
- conformance tests
- reference implementation
- independent verifier
- public test vectors
- VC-shaped credential profile and PROV-shaped graph profile
- SD-JWT/BBS-compatible selective-disclosure profile
- OpenTelemetry GenAI-compatible trace-exchange profile
- source-quality and citation-quality benchmarks
- ownership-attestation and duplicate-claim test vectors

## Conformance Levels

### RDLLM-L0: Replayable Accounting

- payout shares exactly sum to the creator pool
- ledger events replay without hash drift
- unattributed value routes to unattributed escrow rather than a false owner

### RDLLM-L1: Visible Sources

- answer includes source labels and footer
- source references include work IDs and hashes
- external text attribution can credit registered owners even when the answer was
  generated outside the current model route

### RDLLM-L2: Evidence-Continuous Attribution

- every supported claim maps to a source label, chunk ID, evidence text, character
  offsets, and span hash
- span hash prefixes are rendered in the answer footer
- conformance fails if footer span hashes, receipt claims, and ledger claims drift

### RDLLM-L3: Receipted Attribution

- every monetized event emits an attribution receipt
- receipt validates against schema
- receipt hashes prompt, answer, and rendered output
- receipt commits to claim evidence spans
- public receipts support privacy-preserving disclosure

### RDLLM-L4: Transparent and Tamper-Detecting Attribution

- receipt hash is appended to a transparency log
- inclusion proof verifies against a log root
- footer tampering and receipt-payload tampering are detected

### RDLLM-L5: Rights-Enforced Attribution

- policy-denied source matches are blocked from visible citation and normal payout
- rights-conflict escrow is used for traced-but-unlicensed use
- receipts and transparency logs bind policy decisions to the response event
- public reports distinguish unattributed escrow from rights-conflict escrow

### RDLLM-L6: Certified Foundation-Model Attribution

- training cohort manifests are published
- dataset metadata aligns with Croissant/SPDX AI
- source policies expose allowed uses, prohibited uses, jurisdiction, revocation, and
  minimum royalty requirements
- conformance reports are exportable to customers, creators, and regulators
- a certification report passes grounded-generation, external-text, escrow,
  rights-blocked, transparency, tamper-detection, and rights-manifest cases

### RDLLM-L7: Citation-Quality-Certified Attribution

- every event carries a `rdllm-grounding-quality/v1` verdict
- source availability, citation integrity, evidence relevance, fact support, policy
  alignment, and payout alignment are scored separately
- certification fails if paid sources are hidden from the user-visible footer
- certification fails if footer tampering downgrades source support without being
  detected

### RDLLM-L8: Registry-Trust-Certified Attribution

- duplicate or near-duplicate registrations across different creators are detected
  before settlement
- matched works with open ownership conflicts route creator-pool value to
  registry-dispute escrow
- receipts bind registry status, conflict IDs, and registry report hashes to the
  event hash
- certification fails if a disputed claimant is directly paid while the conflict is
  open

### RDLLM-L9: Settlement-Finality-Certified Attribution

- registry-dispute escrow can be released through a signed resolution report
- release reports bind original event hashes, conflict IDs, registry report hashes,
  recipient splits, and balances before and after release
- certification fails if settlement reports are tampered with or no longer
  recompute from the ledger and resolution

### RDLLM-L10: Interoperability-Certified Attribution

- receipts export as VC-shaped attribution credentials with deterministic proofs
- answers export as PROV-shaped graphs linking prompt commitments, output
  commitments, source chunks, supported claims, rights decisions, registry decisions,
  and payout shares
- settlement reports export as VC-shaped escrow-settlement credentials
- certification fails if a bundle hash, credential proof, graph hash, receipt hash,
  or settlement report hash is tampered with

### RDLLM-L11: Attribution-Gap-Certified Attribution

- every response records source-access traces for retrieval and text matching
- attribution-gap reports reconcile accessed, visible, paid, policy-withheld, and
  registry-withheld sources
- certification fails if a consumed source is neither cited, paid, nor escrowed, if
  a footer cites a source that was not accessed, or if a paid source is hidden from
  the user-facing attribution footer

### RDLLM-L12: Selective-Disclosure-Certified Attribution

- receipts carry a `rdllm-selective-disclosure/v1` root and private per-path salts
- public packages disclose source, claim-support, grounding, model, event, rights
  status, registry status, and attribution-gap summary facts
- source quotes, claim evidence text, source-access traces, economics, detailed
  rights decisions, and registry decision internals remain redacted but committed
- certification fails if the public receipt drifts from disclosed leaves, if a
  disclosed value is changed without its salted leaf hash changing, or if the package
  no longer matches the private signed receipt

### RDLLM-L13: Trace-Exchange-Certified Attribution

- receipts carry `rdllm-trace-exchange/v1` commitments for source-access,
  source-reference, and claim-support telemetry
- interop bundles include OpenTelemetry-aligned generation, source-access, citation,
  and claim-support spans
- source-access spans expose stable data-source IDs and structured retrieval
  documents without requiring private prompt or source text disclosure
- certification fails if provider telemetry omits accessed sources, relabels
  citations to different content hashes, or publishes trace summary hashes that no
  longer match the signed receipt

### RDLLM-L14: Royalty-Statement-Certified Attribution

- providers publish `rdllm-royalty-statement/v1` rollups for settlement periods
- statements bind ledger hashes, event roots, share roots, source-access roots,
  source-reference roots, claim-support roots, receipt roots, and trace roots
- creator, escrow, work, and source-usage totals are public while prompts, outputs,
  source quotes, evidence text, and matched text remain private
- certification fails if aggregate payouts do not conserve the creator pool, if
  creator statements drift, if receipt or trace rollups are dropped, or if private
  text appears in the public statement

### RDLLM-L15: Challenge-Correction-Certified Attribution

- creators can file `rdllm-attribution-challenge/v1` reports against a published
  event or royalty statement
- reports bind the original event hash, statement hash, challenged content hash,
  overlap evidence, already-credited status, and non-rewrite remedy
- accepted omissions create corrective claimant payments or rights-conflict escrow;
  already credited sources receive no double payment; weak evidence is rejected
- certification fails if remedy amounts drift, if the report rewrites the original
  event, or if private prompt/source/matched text appears in the public challenge

### RDLLM-L16: Provider-Card-Certified Attribution

- providers publish `rdllm-provider-attribution-card/v1` reports for a model or
  deployment
- cards bind certification level, report hash, source-coverage metrics, supported
  evidence channels, public disclosure surfaces, rights/settlement guarantees,
  challenge policy, limitations, and privacy-safe evidence roots
- cards disclose capability boundaries explicitly: model-internal signals are
  verified through provider telemetry commitments, while raw hidden states, token
  logits, and raw-media decoding internals can remain private
- certification fails if coverage metrics drift from the ledger, if certification
  evidence is stale, or if private prompt/output/source/evidence text appears in the
  public card

### RDLLM-L17: Training-Summary-Certified Attribution

- providers publish `rdllm-training-content-summary/v1` reports for model training
  stages and registered corpora
- summaries align with EU GPAI training-content summaries, Croissant usage policy,
  SPDX AI/Dataset BOMs, and ODRL rights metadata
- summaries bind corpus categories, license counts, allowed/prohibited uses,
  revocation counts, training rights coverage, content-hash roots, policy roots,
  and training-value roots without disclosing work text
- certification fails if training coverage drifts from the corpus, if certification
  or provider-card evidence is stale, or if private work text appears in the public
  summary

### RDLLM-L18: Assurance-Bundle-Certified Attribution

- providers publish `rdllm-assurance-bundle/v1` reports over the public proof pack
- bundles bind artifact names, types, declared hashes, payload hashes, Merkle roots,
  inclusion proofs, publication-profile flags, and issuer signatures
- bundles are privacy-safe: they prove artifact publication without disclosing
  prompt text, answer text, source text, evidence text, work text, or full payloads
- certification fails if artifact hashes drift, inclusion proofs are missing, the
  signature is invalid, or private payloads are exposed

### RDLLM-L19: Answer-Card-Certified Attribution

- providers publish `rdllm-answer-provenance-card/v1` reports for individual
  answers
- cards bind visible source labels, claim span hashes, rendered-output hash,
  receipt hash, trace hash, grounding quality, and attribution-gap verdict
- cards redact prompt text, answer text, source quotes, claim text, evidence text,
  and the full receipt payload
- certification fails if the visible footer is relabeled, if card commitments drift,
  if receipt/trace bindings fail, or if private text appears in the card

### RDLLM-L20: Source-Materialization-Certified Attribution

- providers publish `rdllm-source-verification-report/v1` reports for cited answers
- reports prove cited source hashes, quote hashes, source URIs, claim evidence
  spans, visible footer prefixes, and answer-card bindings resolve to registered
  content
- reports redact prompt text, answer text, source text, source quotes, claim text,
  and evidence text
- certification fails if source hashes are fabricated, answer-card bindings are
  stale, or claim evidence cannot be reproduced from registered content

### RDLLM-L21: Response-Envelope-Certified Attribution

- providers return `rdllm-response-envelope/v1` packages at the API boundary
- envelopes embed the rendered answer, answer provenance card, source verification
  report, public receipt, provider card, and certification posture
- customers can verify footer labels, span prefixes, artifact hashes, answer-card
  binding, source-materialization status, provider disclosure surfaces, and
  certification level without private ledger access
- certification fails if rendered output or embedded public artifacts are tampered
  with

### RDLLM-L22: Integration-Profile-Certified Attribution

- providers publish `rdllm-integration-profile/v1` contracts at a well-known path
- profiles declare generation and verification endpoints, required headers,
  embedded response-envelope artifacts, verifier commands, schemas, public
  surfaces, readiness checks, and bound proof hashes
- platforms, model buyers, creators, and auditors can verify API readiness without
  private ledger access
- certification fails if the API contract drifts, if provider surfaces regress, if
  certification drops below the response-envelope layer, or if bound hashes no
  longer match the provider card, certification report, or response envelope

### RDLLM-L23: Discovery-Manifest-Certified Attribution

- providers publish `rdllm-discovery-manifest/v1` at `/.well-known/rdllm.json`
- manifests catalog public paths for the provider card, certification report,
  integration profile, response envelope, assurance bundle, and training summary
- manifests bind artifact hashes, API contract hashes, schema maps, verifier
  commands, readiness checks, and proof hashes into one customer-verifiable entry
  point
- certification fails if discovery paths drift, if artifact hashes no longer
  reproduce, if provider surfaces regress, if certification falls below the
  integration-profile layer, or if the assurance bundle omits integration-profile
  or response-envelope evidence

### RDLLM-L24: Derivative-Lineage-Certified Attribution

- providers publish `rdllm-lineage-report/v1` for usage events that credit
  derivative works
- reports bind source royalty shares to upstream work IDs, derivative edges,
  content hashes, recursive pass-through obligations, and payout-conservation
  commitments
- summaries, synthetic datasets, transformed corpora, and tool outputs can receive
  direct credit without erasing original upstream owners
- certification fails if upstream works are missing, lineage cycles appear,
  declared upstream hashes drift, obligations no longer conserve source payouts,
  private text leaks, or the report is tampered with

### RDLLM-L25: Provenance-Benchmark-Certified Attribution

- providers publish `rdllm-provenance-evaluation/v1` reports over portable
  attribution-quality benchmarks
- benchmark cases cover clean source reuse, paraphrase, hard decoys, unattributed
  escrow, and derivative-lineage source attribution
- reports expose ranked source decisions, evidence-channel scores, expected recall,
  top-1 accuracy, decoy resistance, escrow accuracy, and grounding verification
  without publishing benchmark prompts, outputs, or source text
- certification fails if benchmark cases are omitted, source rankings drift, report
  hashes or signatures change, or private benchmark/source text leaks

### RDLLM-L26: Counterfactual-Influence-Certified Attribution

- providers publish `rdllm-counterfactual-influence/v1` reports for paid events
- each credited work is removed from the attribution engine, the same prompt and
  answer are replayed, and the report records source-removal status, payout
  reallocation or escrow status, substitute-source pressure, and impact margin
- the public artifact uses work IDs, content hashes, event hashes, scores, and
  commitments instead of prompt, answer, source, quote, or claim evidence text
- certification fails if ablation replay drifts, payout no longer conserves the
  creator pool, source-removal status is wrong, report hashes or signatures change,
  or private event/source text leaks

### RDLLM-L27: Multimodal-Media-Attribution-Certified Attribution

- providers publish `rdllm-media-attribution/v1` reports for image, audio, video,
  3D, and text-shaped media signatures
- each submitted media input is replay-matched against registered assets using
  exact content hashes, perceptual-hash commitments, descriptor-hash commitments,
  ranked scores, and accept thresholds
- matched owners receive payout shares bound to input IDs; weak or unknown media
  influence routes to unattributed-media escrow
- certification fails if media replay drifts, payout no longer conserves the
  creator pool, weak matches are promoted, report hashes or signatures change, or
  raw media, perceptual hashes, or descriptor text leaks

### RDLLM-L28: Model-Internal-Signal-Attribution-Certified Attribution

- providers publish `rdllm-model-signal-attribution/v1` reports for private
  log-probability, activation, gradient, attention, and memorization telemetry
- each report includes an explicit `rdllm-attribution-contract/v1` that names the
  explained output, eligible features, feature granularity, generation process,
  attributed score, held-fixed values, and excluded private internals
- accepted sources receive owner payout shares; weak, denied, or zero-confidence
  signal mass routes to `model_signal_escrow`
- certification fails if telemetry replay drifts, payout no longer conserves the
  creator pool, weak scores are promoted, report hashes or signatures change, the
  attribution contract is unsupported, or raw hidden states, logits, prompts,
  outputs, private traces, or chain-of-thought leak

### RDLLM-L29: Post-Publication-Rights-Remediation-Certified Attribution

- providers publish `rdllm-rights-remediation/v1` reports whenever registered
  works change consent, opt out, revoke rights, change allowed/prohibited uses, or
  update license terms after historical model usage
- each report compares previous and updated policy roots, changed-work hashes,
  historical event references, and future-use enforcement probes
- historical royalty events are preserved by event hash; future denied use is
  blocked or routed to `rights_conflict_escrow`
- certification fails if historical events are rewritten, revoked works remain
  retrievable, future use is allowed despite updated policy, escrow no longer
  conserves the creator pool, report hashes/signatures drift, or work text,
  prompts, outputs, matched text, or private ledger payloads leak

### RDLLM-L30: Semantic-Text-Attribution-Certified Attribution

- providers publish `rdllm-semantic-text-attribution/v1` reports for paraphrased,
  summarized, transformed, or externally generated text
- each submitted output is replay-ranked against registered text with lexical
  overlap, concept overlap, source distinctiveness, sequence support, n-gram
  containment, accept thresholds, and hard-decoy margins
- accepted owners receive payout shares and public footer rows; ambiguous,
  unmatched, or policy-blocked text routes to semantic-text or rights-conflict
  escrow
- certification fails if ranking replay drifts, a hard decoy is paid, creator-pool
